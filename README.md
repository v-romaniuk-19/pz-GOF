# Реалізація GoF патернів на TypeScript

Цей проєкт демонструє практичне застосування класичних патернів проєктування Gang of Four.

## Перелік патернів

### 1. Породжувальні
- **Singleton**: Гарантує наявність лише одного екземпляра класу. Запобігає конфліктам ресурсів.
  
  Проблема: Необхідність мати рівно один екземпляр класу (наприклад, конфігурація або підключення до БД) для всього додатка.
  
  Анти-приклад: Створення нового об'єкта new Database() у кожному модулі, що призводить до витоку пам'яті та зайвих з'єднань.
  
- **Factory Method**: Надає інтерфейс для створення об'єктів, дозволяючи підкласам змінювати тип створюваних об'єктів.
  
  Проблема: Код не повинен залежати від конкретних класів продуктів, які він створює.
  
  Анти-приклад: Величезні блоки switch-case для створення різних типів об'єктів (наприклад, кнопок для Windows/Mac), які доводиться правити при додаванні нового   типу.

### 2. Структурні
- **Adapter**: Дозволяє об'єктам із несумісними інтерфейсами працювати разом.
  
  Проблема: Несумісність інтерфейсів. Ви хочете використати сторонню бібліотеку, але її методи не збігаються з вашим кодом.
  
  Анти-приклад: Переписування всього існуючого коду під нову бібліотеку.
  
- **Decorator**: Дозволяє динамічно додавати об'єкту нову функціональність, обгортаючи його в "обгортку".

  Проблема: Потрібно динамічно додавати обов'язки об'єкту без створення нескінченної кількості підкласів.
  
  Анти-приклад: Створення класів CoffeeWithMilk, CoffeeWithSugar, CoffeeWithMilkAndSugar тощо.

### 3. Поведінкові
- **Strategy**: Визначає сімейство алгоритмів, роблячи їх взаємозамінними прямо під час виконання.

  Проблема: Потрібно вибирати алгоритм під час виконання програми.
  
  Анти-приклад: Один метод pay(), що містить 500 рядків if (type === 'card') ... else if (type === 'paypal').

- **Observer**: Створює механізм підписки, що дозволяє об'єктам стежити за подіями в інших об'єктах.

  Проблема: Потрібно повідомити групу об'єктів про зміни в іншому об'єкті без жорсткої прив'язки.
  
  Анти-приклад: Постійне опитування об'єкта (polling) "чи змінився статус?" кожну секунду.

## Як запустити
1. Встановіть залежності: `npm install`
2. Запустіть демонстрацію: `npx ts-node src/examples/run-all.ts`

## Висновки
Використання патернів дозволяє зробити код:
1. **Гнучким** (легко додавати нові фічі).
2. **Тестованим** (компоненти розділені).
3. **Зрозумілим** для інших розробників, оскільки використовується стандартна "термінологія" архітектури.
